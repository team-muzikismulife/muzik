# 곡 등록 설계 (M2)

"오늘의 곡 한 곡"을 올리는 흐름. 앱의 핵심 쓰기 경로이자, **유일한 쓰기 경로**다.

관련 문서: `유튜브연동설계.md` §2(등록 파이프라인) / `백엔드설계.md` §3(registerTrack) / `frontend.md`(폼·4상태)

---

## 1. 진입점

곡 등록 모달로 들어오는 길은 셋이고, 전부 **내 것일 때만** 열린다.

| 진입점 | 위치 | 조건 |
|---|---|---|
| 오늘의 미션 스트립 | 홈 헤더 아래 | 오늘 곡 미등록 (등록했으면 스트립 비활성) |
| 곡 추가하기 카드 | 홈, 내 팀원 카드 | 내 카드만 누를 수 있다. 남의 빈 카드는 자리만 차지 |
| `⋯` 메뉴 | 홈, 내 곡 카드 | 이미 등록한 경우 → **수정** 모달 |

## 2. 화면 형태 — expo-router 모달 라우트

```
app/room/[id]/track/new.tsx        # 등록  presentation: 'modal'
app/room/[id]/track/[trackId].tsx  # 수정/삭제 (M4)  presentation: 'modal'
```

바텀시트 라이브러리(`@gorhom/bottom-sheet`)를 쓰지 않는다 — reanimated + gesture-handler 의존을
끌고 오는데, Stack의 `presentation: 'modal'`이 iOS 네이티브 시트(스와이프 다운 dismiss)를 공짜로 준다.
URL이 생기므로 뒤로가기·상태복원·(나중에) 유튜브 공유 인텐트 진입도 공짜다.

등록·수정은 **`src/components/TrackForm.tsx` 하나를 `mode: 'create' | 'edit'`로 재사용**한다.

## 3. 파이프라인 — 클라이언트와 서버의 분담

```
[클라이언트]                                    [Cloud Functions: registerTrack]
1. URL 붙여넣기 (클립보드 자동 감지)
2. extractVideoId()      ← 로컬, 즉시
   watch / youtu.be / shorts / embed / live / music
3. fetchVideoMeta()      ← oEmbed, 쿼터 0
   parseTitle + iTunes 보정 → 곡명/가수 편집 가능 (§10)
4. 코멘트 입력 (≤30자)
5. registerTrack(roomId, videoId, comment, title, artist, nickname) ─→ 6. 멤버 검증
                                                7. **서버 시각으로 dateKey 확정**
                                                   (클라는 dateKey를 보내지 않는다)
                                                8. videos.list 검증 5규칙
                                                9. tracks/{uid}_{dateKey} **create**
                                                   (이미 있으면 already-exists)
                                               10. days upsert
   ←─────────────────────────── track 스냅샷 ──┘
11. 모달 닫기 → 홈 리스트에 즉시 반영 (onSnapshot)
```

**클라이언트가 하지 않는 것**
- `dateKey` 계산해서 보내기 → 서버 시각이 유일 기준 (시계 조작 방지)
- `thumbnailUrl` 만들어 보내기 → `videoId`에서 파생한다 (설계 결정 #8)
- 하루 1곡 최종 판정 → 서버의 `create`가 원천 차단. 클라는 낙관적 차단(UI)만.

**서버 검증 5규칙** (유튜브연동설계 §2)
1. `privacyStatus != 'public'` → 거부 "비공개/삭제된 영상"
2. `regionRestriction`에 KR 차단 → 거부
3. `liveBroadcastContent != 'none'` → 거부 (라이브는 마감 후 재생 불가)
4. `embeddable == false` → **등록은 허용**, `embeddable: false` 저장 (미리듣기만 스킵 + "유튜브 전용" 배지)
5. `duration` → `durationSec` 저장

## 4. 상태 머신

```
idle ──(URL 입력)──→ parsing ──(파싱 성공)──→ fetching ──(oEmbed 성공)──→ preview
  ↑                     │                        │                          │
  │                  (파싱 실패)              (oEmbed 404)              (등록 탭)
  │                     ↓                        ↓                          ↓
  └──────────────── inputError ←────────────────┘                     submitting
                                                                           │
                                              ┌────────────────────────────┤
                                       (already-exists)                 (성공)
                                              ↓                            ↓
                                     "수정하기"로 분기                  모달 닫기
```

- **`preview` 전에는 등록 버튼이 비활성**이다. 곡을 확인하지 않고 올릴 수 없다.
- `submitting` 중에는 버튼을 잠근다 — 더블탭하면 두 번째가 `already-exists`를 맞는다.

## 5. 검증 (zod, `src/schemas/`)

클라이언트와 Functions가 **같은 스키마를 공유**한다. 검증 규칙이 갈라지는 것을 막는다.

| 필드 | 스키마 | 규칙 |
|---|---|---|
| videoId | `VideoIdSchema` | `/^[\w-]{11}$/` |
| comment | `CommentSchema` | trim, ≤30자 (빈 문자열 허용) |
| title | `TitleSchema` | trim, 1~80자 (**사용자 편집 가능** → 검증 대상) |
| artist | `ArtistSchema` | trim, 1~80자 (**사용자 편집 가능** → 검증 대상) |
| roomId | — | 비어있지 않음 |

에러 메시지는 **스키마에 담는다.** 화면은 문구를 만들지 않고 받아서 보여주기만 한다.
`title`/`artist`는 자동 추정값을 사용자가 고칠 수 있으므로(§10) **신뢰할 수 없는 입력**이다 —
길이 검증을 클라(제출 전)·rules·(정본)Functions에서 모두 건다.

## 6. 에러 처리 — `already-exists`는 에러가 아니다

| 상황 | 코드 | 처리 |
|---|---|---|
| 유튜브 링크가 아님 | (로컬) | 인풋 하단 인라인: "유튜브 링크가 아닌 것 같아요" |
| 없는/비공개 영상 | (oEmbed 404) | "없는 영상이거나 비공개예요" |
| **오늘 이미 올림** | `already-exists` | **막다른 에러로 두지 않는다.** "오늘은 이미 곡을 올렸어요" + **[내 곡 수정하기]** CTA로 전환 |
| 비공개·지역차단·라이브 | `invalid-argument` | 서버가 준 사유 문구를 그대로 노출 |
| 팀원이 아님 | `permission-denied` | "이 팀의 팀원이 아니에요" → 홈으로 |
| 오프라인 | `unavailable` | 토스트 "네트워크 연결 후 다시 시도" + **입력값 유지** (다시 치게 하지 않는다) |
| 코드 충돌·쿼터 | `resource-exhausted` | "잠시 후 다시 시도해 주세요" |

매핑은 `src/lib/errors.ts` 한 곳에서 한다.

## 7. 엣지케이스

| 케이스 | 처리 |
|---|---|
| **03:59:59 등록 vs 04:00 롤오버** | 서버 시각이 유일 기준. 응답의 `dateKey`로 UI 교정 |
| 더블탭 (동시 등록) | 제출 중 버튼 잠금 + 서버 `create`가 최종 방어 |
| 붙여넣기에 잡파라미터 (`?si=`, `&t=`) | `extractVideoId`가 무시 |
| Shorts / music.youtube / youtu.be | `extractVideoId`가 전부 지원 |
| videoId 11자를 그대로 붙여넣음 | 지원 (URL이 아니어도 됨) |
| oEmbed가 느림 | 미리보기 자리에 스켈레톤. 등록 버튼은 미리보기 전까지 비활성 |
| `embeddable == false` | 등록되고 "유튜브 전용" 배지. 미리듣기 큐에서만 스킵 |
| 오프라인에서 미리보기 시도 | oEmbed 실패 → "네트워크 연결 후 다시 시도" (입력값 유지) |

## 8. 지금 구현하는 범위 (M2-a) vs 나중 (M2-b)

**Firebase가 아직 없다(M1 미착수).** 그래서 전송 경계를 `src/lib/api.ts`로 끊고,
클라이언트 파이프라인은 **전부 진짜로** 만든다.

| | M2-a (지금) | M2-b (M1 이후) |
|---|---|---|
| URL 파싱 | ✅ `extractVideoId` (실제) | 그대로 |
| 미리보기 | ✅ `fetchVideoMeta` oEmbed (실제 네트워크) | 그대로 |
| 코멘트 검증 | ✅ zod (실제) | 그대로 + 서버 재검증 |
| 모달·상태·에러 UI | ✅ 전부 | 그대로 |
| **`registerTrack`** | `src/lib/api.ts` → **로컬 store 기록** | → **Functions callable** 로 교체 |
| dateKey 확정 | 클라 `todayKey()` (임시) | **서버 시각** |
| videos.list 5규칙 | ❌ (oEmbed만) | ✅ |
| 실시간 반영 | zustand store | onSnapshot |

**교체 지점은 `src/lib/api.ts` 한 파일이다.** 화면·폼·에러 처리는 손대지 않는다.

`embeddable`은 oEmbed로 알 수 없으므로 M2-a에선 `true`로 낙관 저장하고, 미리듣기의 `onError`가
방어한다(유튜브연동설계 §4의 쿼터 초과 저하 모드와 **같은 코드 경로**).

## 10. 제목 정리 · 붙여넣기 개선 (사용성 보강)

oEmbed는 `title`(영상 제목)·`author_name`(채널명)만 준다. 무가공 저장하면 가수가 "HYBE LABELS"
같은 채널명으로 들어가 이상해진다. 3단으로 다듬은 뒤 **편집 가능 필드**로 노출한다.

### 10-1. 곡명/가수 정리 — 휴리스틱 → iTunes → 편집

```
oEmbed(title, author) ──→ parseTitle()  ──→ itunes.searchTrack()  ──→ 편집 가능 입력
   원본 제목/채널명        ①휴리스틱 분리        ②텍스트 보정             ③사용자가 최종 확정
```

1. **`parseTitle(rawTitle, author)`** (`src/lib/youtube.ts`) — 순수 함수, 즉시.
   - 따옴표로 감싼 곡명(`Artist 'Song' MV`) 우선 추출 → 구분자 분리(`Artist - Song`) → 폴백(채널명=가수).
   - 홍보 접미사·괄호 노이즈(`Official MV`, `[MV]`, `(Lyrics)`, `(가사)` …), 채널명 꼬리(`- Topic`,`VEVO`) 제거.
2. **`itunes.searchTrack(query)`** (`src/lib/itunes.ts`) — best-effort, 무키·무쿼터.
   - iTunes Search API로 정식 `trackName`/`artistName`을 받아 초기값을 덮는다.
   - 실패·무결과·비카탈로그 영상이면 `null` → 휴리스틱 값 유지. 등록을 **막지 않는다**.
   - **⚠️ 불변 원칙**: iTunes는 **텍스트 보정 전용**. `artworkUrl`은 가져오지도 저장하지도 않는다.
     썸네일은 계속 `videoId` 파생(설계결정 #8) — iTunes 아트를 따로 두면 videoId와 어긋난다.
3. **편집 가능 필드** (`VideoPreview.tsx`) — 곡명·가수를 `TextInput`으로. 자동 추정이 틀려도 한 탭 수정.
   경합은 `reqId` latest-wins로 방어(빠른 입력 시 이전 oEmbed/iTunes 응답 폐기).

실측(2026-07): `NewJeans (뉴진스) 'Ditto' Official MV` / `HYBE LABELS` → **Ditto / 뉴진스**.

### 10-2. 붙여넣기 — 클립보드 자동 감지

- 모달 마운트 시 `Clipboard.hasStringAsync()`→`getStringAsync()`→`extractVideoId`로 유튜브 링크면
  **"복사한 링크 붙여넣기" 칩**을 원탭 노출(`TrackForm.tsx`). 기존 붙여넣기 버튼은 유지.
- **웹은 자동 감지를 건너뛴다** — 브라우저 `clipboard.readText`가 사용자 제스처를 요구할 수 있어
  `Platform.OS==='web'`이면 버튼만 쓴다(자동 스니핑 실패해도 조용히 무시).

### 10-3. '유튜브에서 공유' 프리필 훅 (M5 준비)

- `track/new` 라우트가 `?url=` 쿼리를 받으면 `TrackForm`을 프리필한다(`initialUrl`).
- 실제 공유 진입(네이티브 Share Extension / 웹 PWA Share Target)은 **dev-client 빌드가 필요** →
  M5로 미룬다. 지금은 파라미터 수신부만 배선(진입점이 이 훅으로 수렴하도록).

### 10-4. 검증 (추가)
- [ ] 지저분한 MV 링크 → 곡명/가수가 정식으로 채워지고 **편집 가능**
- [ ] 비카탈로그 영상(랜덤 클립) → 휴리스틱/채널명 폴백, 여전히 편집 가능
- [ ] 곡명/가수를 비우고 제출 → 스키마 에러(1~80자)
- [ ] 유튜브 링크 복사 후 모달 열기(네이티브) → 칩 노출 → 원탭 프리필
- [ ] `?url=` 로 진입 → 폼 프리필

## 11. 수정 · 삭제 + 날짜 탭 (M4)

정본 계획은 `updateTrack`/`deleteTrack` **Functions**지만, 데모(Spark)에선 registerTrack과 같은
**클라 직접 트랜잭션**(`src/lib/api.ts`)으로 구현한다. 대상은 언제나 `{uid}_{오늘}` — 지난 날짜 곡은
문서 ID가 오늘 것이 아니라 **애초에 대상이 될 수 없다**(당일·본인 강제를 ID가 준다).

### 11-1. 수정 — 단일 모달 재사용
- **별도 라우트를 만들지 않는다.** `track/new.tsx`가 오늘 내 곡을 찾으면(`tracks.find(uid & dateKey==오늘)`)
  `editTrack`을 `TrackForm`에 넘겨 **수정 모드**로 전환한다. ⋯ 메뉴·미션 스트립·AddTrackCard가 모두
  이 한 모달로 수렴한다(진입점은 그대로, 모드만 자동).
- 수정 모드는 저장값(videoId/title/artist/comment)을 시드하고 미리보기를 바로 `ready`로 —
  **자동 파싱·iTunes·클립보드 감지를 돌리지 않는다**(사용자가 고쳐둔 값을 덮지 않기 위해).
  URL을 바꾸면 그때 새 곡으로 재파싱된다.
- `updateTrack`: videoId·메타·comment만 갱신하고 uid·nickname·dateKey·order·createdAt은 보존.
  videoId가 바뀌고 그 곡이 그날 커버였으면 `days.coverVideoId`도 따라간다.

### 11-2. 삭제 — 인라인 2단 확인
- **`Alert` 다중 버튼을 쓰지 않는다** — react-native-web에서 불안정. 대신 삭제 버튼을 **두 번 눌러** 확정
  ("곡 삭제" → "한 번 더 누르면 삭제돼요")한다. 웹·네이티브 모두 동일 동작.
- `deleteTrack`: 트랜잭션으로 track 삭제 + `days.trackCount-1`, **0이면 days 문서 삭제**(빈 날짜 탭이
  영원히 남지 않게). 지운 곡이 커버였고 남은 곡이 있으면 order 최소 곡으로 **커버 재지정**
  (트랜잭션은 쿼리 불가 → 커밋 후 별도 쿼리, 실패해도 삭제엔 영향 없음).

### 11-3. 날짜 탭 — 실데이터
- 홈이 `days` 구독(`subscribeDays`, 문서ID desc 14개)을 room store에 싣고, 날짜 탭을
  **곡이 있는 날짜 + 오늘**로 구성한다(오름차순). 빈 날짜는 뜨지 않는다.
- 과거 탭 탭 → `playlist/[dateKey]`가 그 날짜를 구독(§M3). 빈 날짜로 직접 진입해도 빈 상태로 안전.
