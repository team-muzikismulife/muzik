# 온보딩 구현 계획 (이슈 #14)

대상 화면: `app/index.tsx` — Figma **OnBoarding `107:1126`**
범위: **① 익명 로그인 ② 참여 중인 팀 목록 ③ 새로운 팀 개설**
전제: 마일스톤 M1의 앞부분. 초대 코드 입장(`joinRoom`)·딥링크는 후속 이슈.

---

## 1. 설계 문서에 없던 것 — 이번에 정한 결정

온보딩을 실데이터로 붙이려는 순간 기존 설계에 구멍 셋이 드러났다. 결정과 근거를 남긴다.

### 결정 1. 팀 목록은 **서버가 진실**이다 — `members.uid` + collectionGroup

`rooms/{roomId}/members/{uid}`는 서브컬렉션이라 "uid가 나인 방 전부"를 뽑을 수 없었다.
`백엔드설계.md` §7의 최초 계획은 **"방 리스트 = 로컬 백업 코드 기반"**(AsyncStorage)이었는데, 이 안을 버린다.

- **버린 이유**: 앱 삭제·기기 변경 시 팀 목록이 통째로 사라진다. 익명 uid가 소멸하는 건 어차피 MVP가
  수용한 한계지만(§1), 거기에 **"uid는 살아 있는데 목록만 없는"** 두 번째 실패 모드까지 얹을 이유가 없다.
- **채택안**: `members` 문서에 `uid` 필드를 실어 두고(문서 ID와 중복이지만 **쿼리 가능한 형태**로),
  `collectionGroup('members').where('uid','==',myUid)`로 조회한다.
  `src/types/models.ts`의 `Member`에는 이미 `uid`가 있다 — 스키마 문서만 이를 빠뜨리고 있었다.
- **대가**: collectionGroup 인덱스 1개 + Security Rules에 collectionGroup용 read 룰 1줄이 추가된다.

### 결정 2. **"새 기록" 배지는 M2로 미룬다**

배지 조건은 "이 방의 마지막 곡 등록 시각 > 내가 마지막으로 본 시각"인데, `rooms`에 `lastTrackAt`이 없다.
이 필드는 **`registerTrack`이 곡을 쓸 때 같이 갱신해야** 의미가 생긴다(M2). 곡이 하나도 없는 M1 시점에는
어차피 항상 false다. 지금 필드만 추가해 봐야 쓰는 쪽이 없는 죽은 스키마가 된다.

→ M1에서는 `TeamCard`의 `hasNew`를 넘기지 않는다(배지 미표시). Figma 세 번째 카드가 이미 이 상태를 정의해 뒀다.
→ M2에서 `rooms.lastTrackAt`(서버) + `session.lastVisitedAt[roomId]`(로컬) 비교로 되살린다.

### 결정 3. 헤더 닉네임 — **마지막에 쓴 닉네임을 로컬에 기억한다**

닉네임은 **방별 속성**이다(`백엔드설계.md` §1). 그래서 "환영합니다, ○○님"에 쓸 전역 닉네임이 없다.

- `session` 스토어에 `lastNickname`을 AsyncStorage로 영속시킨다. 팀 개설·입장 폼의 **기본값**으로도 쓴다.
- 아직 아무 방에도 안 들어간 신규 사용자는 닉네임이 없다 → 헤더를 **"환영합니다"**로만 렌더한다
  (현재 코드가 이미 이 분기를 갖고 있다).

### 결정 4. 팀에 **이모지 속성을 두지 않는다**

홈 시안(`107:569`)의 헤더는 "🎵 무직은 내 삶", 온보딩 대안 시안(`107:703`)도 팀마다 이모지(🎵🎸🎓)를 달고 있다.
하지만 **정본 온보딩(`107:1126`)에는 이모지가 없고**, `rooms` 스키마에도 해당 필드가 없다.

→ **이모지를 뺀다. 팀 이름만 쓴다.** 이모지를 살리려면 `rooms.emoji` 필드 + 개설 시 이모지 선택 UI가 따라붙는데,
   MVP에서 그만한 값을 하지 못한다. (필요해지면 그때 필드를 추가한다 — 지금 넣으면 쓰는 쪽 없는 죽은 스키마다.)

### 결정 5. 헤더의 팀 드롭다운(⌄) = **온보딩으로 돌아가기**

홈 시안의 헤더에 `chevron-down`이 있으나 **드롭다운이 열린 시안은 Figma에 없다**(파일 전체 검색 결과 시트·메뉴·팝오버 프레임 없음).
즉 누를 곳만 있고 눌렀을 때 나올 것이 없는 상태다.

→ M1에서는 `⌄` 탭 시 **온보딩(`/`)으로 이동**한다. 온보딩이 곧 "참여 중인 팀 목록"이므로 팀 전환이 성립한다.
   새 화면도 새 디자인도 필요 없다.
→ 바텀시트 스위처가 정석 UX지만(Slack 워크스페이스 스위처), 시안이 없어 디자인 확정을 기다려야 한다.
   M1의 목표("두 기기가 같은 방에 있다")에 팀 전환은 필요 없으므로 **같은 데이터 레이어를 쓰는 시트로 나중에 승격**한다.

## 2. Firestore 스키마 변경

```diff
  rooms/{roomId}/members/{uid}
-   nickname, joinedAt, photoColor
+   uid, nickname, joinedAt, photoColor      # uid: 문서 ID와 중복이나 collectionGroup 쿼리에 필요
```

**인덱스 추가**: collectionGroup `members` — `uid ASC`

**Security Rules 추가**:

```
// 내 멤버 문서는 방을 몰라도 읽을 수 있어야 한다 (참여 중인 팀 목록 조회)
match /{path=**}/members/{memberId} {
  allow read: if request.auth != null && resource.data.uid == request.auth.uid;
}
```

> 룰이 `resource.data.uid`를 보므로, 클라이언트 쿼리도 반드시 `where('uid','==',myUid)`로 **같은 제약**을
> 걸어야 통과한다. 이 둘이 어긋나면 `permission-denied`가 나는데 원인이 룰인지 쿼리인지 헷갈린다.

## 3. 데이터 흐름

```
앱 시작
  └ signInAnonymously()  → uid (AsyncStorage 영속)
        └ collectionGroup('members').where('uid','==',uid)   → 내 멤버 문서 N개
              └ 각 문서의 부모 roomId 추출 → rooms/{roomId} 문서 N개 병렬 get
                    └ 각 방의 members limit 4 get (아바타 스택용)
                          → TeamCard[]  (이름, memberCount, 아바타 4개)
```

- "N명 참여중"은 `rooms.memberCount`를 그대로 쓴다 (멤버 전체를 읽지 않는다).
- 아바타 스택은 방마다 최대 4명만 읽는다 (`TeamCard`의 `MAX_AVATARS = 4`, Figma와 일치).
- 온보딩은 **포커스 시 1회 조회**로 충분하다. 실시간 구독(onSnapshot)은 방 홈(members)에서 한다 — 온보딩에서
  방 N개를 실시간 구독하면 읽기 비용이 방 수에 비례해 늘어난다.

| # | 작업 | 산출물 |
|---|---|---|
| P1 | Firebase 초기화 + 에뮬레이터 접속 | `src/lib/firebase.ts`, `src/lib/emulator.ts` |
| P2 | 익명 로그인 부트스트랩 + 세션 스토어 | `src/store/session.ts`, `app/_layout.tsx` (스플래시 게이팅) |

### P1의 함정 (백엔드설계.md §7 — 미리 알고 들어간다)

| 함정 | 대응 |
|---|---|
| **RN Auth 영속** — `getAuth()`는 재시작 시 로그아웃. **uid 소멸 = 방 소유권 소멸** | `initializeAuth(app, { persistence: getReactNativePersistence(AsyncStorage) })` |
| Expo Go 실기기는 `localhost`를 못 본다 | `Constants.expoConfig.hostUri`에서 LAN IP 파싱 → `connect*Emulator` |
| Functions 리전 누락 → us-central1 호출 → 정체불명 `internal` | `getFunctions(app, 'asia-northeast3')` |
| Expo Go에서 onSnapshot이 조용히 멈춤 | `experimentalForceLongPolling` |

### P4 — 화면은 조립만 (frontend.md)

- 컴포넌트에서 Firestore를 직접 부르지 않는다. `src/lib/db.ts`(읽기) / `src/lib/api.ts`(쓰기)를 통한다.
- 4상태 전부 구현: 로딩(Skeleton) / 성공 / **빈 목록**(참여 중인 팀 없음 → 개설 유도) / 에러(재시도).
  빈 상태는 신규 사용자의 **첫 화면**이다 — 가장 자주 보게 될 상태다.
- `TeamCard`·`Avatar`·`StateView`는 이미 있다. 새로 만들지 않는다.

## 5. 완료 기준

1. 앱을 껐다 켜도 **같은 uid**로 복원된다 (Auth 영속 검증).
2. "새로운 팀 개설하기" → 팀 이름·닉네임 입력 → 팀이 생성되고 **초대 코드**가 나온다.
3. 앱 재시작 후에도 개설한 팀이 **목록에 남아 있다** (collectionGroup 조회 검증).
4. 참여 중인 팀이 없으면 빈 상태가 뜬다 (흰 화면이 아니다).
