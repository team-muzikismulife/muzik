# 곡 검색 설계 (M2-c)

앱 안에서 곡을 검색해 등록하는 경로. 지금은 URL 붙여넣기뿐이라 사용자가
"유튜브 앱 → 검색 → 공유 → 우리 앱" 왕복을 해야 한다.

관련 문서: `곡등록설계.md`(등록 파이프라인 정본) / `유튜브연동설계.md` §1·§4(연동 수단·쿼터) / `백엔드설계.md` §3(Functions)

---

## 1. 대원칙 — 검색은 videoId를 얻는 두 번째 방법일 뿐이다

**저장 스키마는 바뀌지 않는다.** 검색이 하는 일은 `registerTrack(roomId, videoId, comment)`에
넘길 `videoId` 하나를 알아내는 것뿐이고, 그 뒤는 기존 파이프라인(`곡등록설계.md` §3)을 그대로 탄다.
서버 dateKey 확정도, 검증 5규칙도, `{uid}_{dateKey}` create도 손대지 않는다.

따라서 **검색에서 얻은 어떤 메타데이터도 DB에 저장하지 않는다.** 곡명·아티스트·앨범아트는
전부 화면에 그리고 버린다. videoId만 영구 원본이라는 설계결정 #8이 검색 경로에도 그대로 적용된다.

---

## 2. iTunes를 쓰는 진짜 이유 — 앨범아트가 아니라 쿼터 게이팅

유튜브 `search.list`는 **100유닛/호출**이다. `videos.list`(1유닛)의 100배다.
일일 기본 쿼터가 10,000이므로 **타이핑할 때마다 유튜브를 검색하면 하루 100번 만에 앱 전체가 멈춘다.**
자동완성에 붙이는 건 논외고, 곡당 1회로 제한해도 하루 100곡이 상한이다.

iTunes Search API는 인증·API 키·쿼터가 없다. 그래서 이렇게 나눈다.

| 구간 | 수단 | 비용 |
|---|---|---|
| 사용자가 타이핑하는 동안 (호출 N회) | iTunes Search | **0** |
| 사용자가 곡을 확정한 순간 (호출 1회) | 유튜브 `search.list` | 100유닛 |

**iTunes가 타이핑 구간 전체를 흡수하고, 유튜브 호출은 사용자가 곡을 고른 뒤 딱 한 번만 일어난다.**
앨범아트가 예쁜 건 부수 효과지 도입 이유가 아니다.

## 3. 파이프라인

```
[클라이언트]                                  [Functions: resolveVideoId]
1. 검색어 입력 (debounce 300ms)
2. iTunes Search ─── 쿼터 0, 매 입력마다 OK
   term={q}&entity=song&country=kr&limit=20
   → trackId·trackName·artistName
     ·artworkUrl100·trackTimeMillis
3. 결과 리스트 렌더 (앨범아트 600x600)
4. 곡 탭 → resolveVideoId(itunesTrackId, ──→ 5. resolutions/{itunesTrackId} 조회
   trackName, artistName, durationSec)         ├ HIT  → videoId[] (0유닛)
                                               └ MISS → search.list 1회 (100유닛)
                                            6. 후보에 videos.list 배치 1회 (1유닛)
                                               → 라이브·비공개 제거, duration 대조
                                            7. resolutions 캐시에 **videoId만** 저장
   ←────────── 후보 3~5개 (videoId,          ┘
                title, channel, durationSec)
8. 후보 카드 선택 → preview 확정
9. 코멘트 입력 → registerTrack(roomId, videoId, comment)
                                            → 기존 등록 파이프라인 그대로
```

`4`에서 `registerTrack`이 아니라 별도 함수로 끊는 이유: 후보를 **사용자에게 보여주고 고르게** 해야 하고,
그러려면 해결과 등록 사이에 사용자의 결정이 한 번 들어가야 한다.

---

## 4. 후보 자동선택을 하지 않는다

`search.list` 최상단이 공식 음원이라는 보장이 없다. 커버·라이브·8시간 루프·리액션·시간 지연 믹스가 온다.
자동선택은 두 가지로 실패한다.

1. **신뢰 손실.** 엉뚱한 영상이 팀 플레이리스트에 박히면 사용자는 다시 검색을 안 쓴다.
2. **등록 실패.** 라이브 영상이 뽑히면 서버 검증 5규칙 중 3번(`liveBroadcastContent != 'none'` 거부)에
   걸려서, 사용자 입장에선 "검색해서 골랐는데 등록이 안 되는" 이해 불가한 에러가 된다.

그래서 `search.list` **1회로 받은 상위 5개를 후보로 제시하고 사용자가 고른다.** 추가 쿼터는 0이다
(`maxResults=5`는 1개일 때와 같은 100유닛).

`곡등록설계.md` §4의 "`preview` 전에는 등록 버튼이 비활성 — 곡을 확인하지 않고 올릴 수 없다"와 같은 규칙이다.
**후보 선택이 검색 경로의 `preview`다.**

### 정렬 가중치

`videos.list`로 받은 메타로 후보를 재정렬한다. 유튜브 관련도 순서를 그대로 믿지 않는다.

| 신호 | 처리 |
|---|---|
| `liveBroadcastContent != 'none'` | **제거** (서버 검증에서 어차피 거부됨 — 보여주면 안 됨) |
| `privacyStatus != 'public'` | **제거** |
| duration이 iTunes `trackTimeMillis` ±15초 | **가중치 상향** — 공식 음원일 확률이 가장 높은 단일 신호 |
| duration이 iTunes의 2배 초과 | 하향 (루프·앨범 전체·믹스) |
| 채널명이 아티스트명 포함 / `- Topic` 접미 | 상향 (유튜브 자동 생성 아티스트 채널) |

## 5. 캐시 — `resolutions/{itunesTrackId}`

```
resolutions/{itunesTrackId}
  videoIds: string[]   // 후보 (정렬된 순서)
  resolvedAt: number
```

**videoId만 저장한다. title·channel·artwork는 저장하지 않는다.** 두 가지를 동시에 산다.

1. **30일 정책 회피.** YouTube API로 받은 데이터는 30일 내 갱신/삭제 의무가 있다(`유튜브연동설계.md` §6).
   videoId는 영구 원본이라 대상이 아니다. 메타를 캐시하면 이 캐시에도 lazy refresh를 붙여야 하는데,
   그럴 이유가 없다.
2. **설계결정 #8과 일관.** 저장하면 어긋난다. 어긋났었다.

캐시 히트 시에도 후보 메타(title·channel·duration)는 `videos.list` 1회(**1유닛**)로 새로 받는다.
100유닛이 1유닛이 되는 것이고, 이게 이 캐시의 전부다.

**히트율이 높은 이유**: 같은 곡을 여러 팀·여러 사람이 올린다. 차트 상위곡일수록 더 그렇다.
캐시 키가 팀 단위가 아니라 **전역**이라(`rooms/` 밖의 최상위 컬렉션) 앱 전체가 한 번의 해결을 공유한다.
곡당 `search.list` 비용이 평생 1회로 수렴한다.

캐시는 만료시키지 않는다. 곡 → 공식 음원 videoId 매핑은 변하지 않는다. 영상이 삭제되면
기존 `unavailable` 처리(`유튜브연동설계.md` §5)가 등록 이후 경로에서 잡는다.

## 6. 쿼터 재계산 (§4 표에 반영)

| 시나리오 | 소비 |
|---|---|
| 검색 → 곡 확정, **캐시 미스** | search.list 100 + videos.list 1 = **101유닛** |
| 검색 → 곡 확정, **캐시 히트** | videos.list 1 = **1유닛** |
| 타이핑 (iTunes) | **0** |
| 최종 등록 (registerTrack 검증) | videos.list 1유닛 |

**최악(히트율 0%): 일 99곡.** 초기에는 여기에 가깝다. 히트율이 오르면 상한이 수천 곡으로 올라간다.

### 저하 모드 (킬스위치)

쿼터가 이 기능 하나 때문에 앱 전체(등록 검증 포함)를 태우는 걸 막아야 한다.
기존 핸드오프 킬스위치(`유튜브연동설계.md` §3-3)와 **같은 패턴**을 쓴다.

```
config/app
  searchMode: 'on' | 'off'    // 앱 시작 시 1회 구독 (handoffMode와 같은 문서)
```

- `off` → 검색 탭을 숨기고 링크 붙여넣기만 노출. **등록 기능 자체는 살아 있다.**
- Functions가 일일 `search.list` 카운터를 보고 임계치(예: 80회) 초과 시 `off`로 자체 전환.
  배포 없이 문서 수정만으로 복구·차단 양방향 가능.

검색이 죽어도 앱이 죽지 않는 구조여야 한다. **그래서 URL 붙여넣기 경로를 없애지 않는다.**

---

## 7. URL 붙여넣기 경로는 유지한다

검색이 대체가 아니라 추가인 이유:

- 검색에 안 나오는 곡 — 믹스, 라이브 세션, 개인 업로드, 아직 iTunes에 없는 신곡, 유튜브에만 있는 곡
- 유튜브 공유 인텐트 진입(`곡등록설계.md` §2)은 URL을 들고 들어온다
- `searchMode: 'off'` 저하 모드의 착지점

등록 모달(`TrackForm`)에 **[검색] | [링크]** 토글을 둔다. 기본값은 검색.

---

## 8. 화면

```
app/room/[id]/track/new.tsx     # 기존 등록 모달 — 상단에 [검색]|[링크] 토글 추가
```

새 라우트를 만들지 않는다. 검색·후보 선택·코멘트가 **한 모달 안에서 단계로 흐른다** —
등록은 하나의 행위지 화면 세 개가 아니다.

```
검색 입력 → iTunes 결과 리스트 → (곡 탭) → 후보 3~5개 → (후보 탭) → preview + 코멘트 → 등록
                                    ↑                        │
                                    └──────── [다른 영상 고르기] ┘
```

`preview`에서 **[다른 영상 고르기]로 후보 리스트에 돌아갈 수 있어야 한다.** 잘못 고른 걸
되돌리는 유일한 길이 모달을 닫고 처음부터 하는 것이면 아무도 안 쓴다.

썸네일: iTunes 앨범아트는 **1:1**, 유튜브 썸네일은 **16:9**다. 섞어 쓰지 않는다.
- iTunes 결과 리스트 → 앨범아트 1:1 (`artworkUrl100`의 `100x100bb` → `600x600bb` 치환)
- 후보 리스트·preview·등록된 곡 카드 → **`YoutubeArt` 16:9** (videoId 파생, 폴백 체인 — `유튜브연동설계.md` §5-1)

곡 카드 배경은 지금도 앞으로도 유튜브 썸네일이다. iTunes 아트워크는 검색 리스트에서만 살고 저장되지 않는다.

## 9. 검증 (zod, `src/schemas/`)

| 필드 | 규칙 |
|---|---|
| `q` (검색어) | trim, 1자 이상. 빈 문자열이면 호출하지 않는다 |
| `itunesTrackId` | 양의 정수 |
| `trackName` / `artistName` | trim, 비어있지 않음 (search.list 쿼리 조립에 들어감) |
| `durationSec` | 선택 — iTunes `trackTimeMillis`가 없는 항목이 있다 |

`resolveVideoId` 출력도 스키마로 검증한다. 유튜브 응답 형태를 Functions 밖에서 신뢰하지 않는다.

## 10. 에러 처리

매핑은 기존대로 `src/lib/errors.ts` 한 곳에서 한다.

| 상황 | 코드 | 처리 |
|---|---|---|
| iTunes 결과 0건 | — | 빈 상태 + **"링크로 등록하기" CTA** (막다른 길로 두지 않는다) |
| iTunes 네트워크 실패 | — | 인라인 "검색에 실패했어요" + 재시도. **링크 탭은 살아 있다** |
| 후보가 0개 (전부 라이브/비공개) | `not-found` | "유튜브에서 이 곡을 찾지 못했어요" + 링크 탭 유도 |
| search.list 쿼터 초과 | `resource-exhausted` | 검색 탭 비활성 + 링크 탭 유도 (`searchMode: 'off'`와 같은 착지점) |
| 이미 오늘 곡 등록함 | `already-exists` | 기존과 동일 — **[내 곡 수정하기] CTA** (`곡등록설계.md` §6) |
| 팀원 아님 | `permission-denied` | 기존과 동일 |
| 오프라인 | `unavailable` | 토스트 + **입력값 유지** |

**모든 검색 실패의 착지점이 "링크로 등록하기"다.** 검색은 지름길이지 유일한 길이 아니다.

## 11. 엣지케이스

| 케이스 | 처리 |
|---|---|
| 타이핑 중 이전 요청이 늦게 도착 | 요청 시퀀스 번호로 stale 응답 폐기 (debounce만으론 부족) |
| iTunes에 `trackTimeMillis` 없음 | duration 가중치만 스킵. 나머지 정렬 신호로 진행 |
| iTunes 아트워크 404 | 플레이스홀더. 어차피 저장 안 하므로 영향은 그 화면 한정 |
| **웹에서 iTunes CORS** | **문제 없음 (2026-07-17 실측).** §11-1 참고 — JSONP·프록시 전부 불필요 |
| iTunes 응답 `content-type: text/javascript` | JSON인데 헤더가 `text/javascript`다(2026-07-17 실측). 브라우저 `response.json()`은 content-type을 안 보므로 무해. **content-type으로 파싱을 결정하는 클라이언트를 쓰지 말 것** — 반드시 텍스트로 받아 파싱하거나 `.json()`을 쓴다 |
| 같은 곡을 두 사람이 동시에 첫 해결 | resolutions에 둘 다 write — 같은 값이라 무해. 트랜잭션 불필요 |
| 후보 고른 뒤 등록에서 서버 검증 거부 | 후보 리스트로 되돌리고 사유 노출 (§8의 [다른 영상 고르기] 경로 재사용) |

## 11-1. iTunes Search API 실측 (2026-07-17)

착수 전 확인이 필요했던 항목들. **전부 통과 — 우회 설계가 필요 없다.**

| 항목 | 결과 |
|---|---|
| **CORS** | **통과.** 요청 `Origin`을 `access-control-allow-origin`에 그대로 반사한다. OPTIONS preflight도 200 (`allow-methods: GET`, `allow-headers: *`, `max-age: 86400`). **JSONP·Functions 프록시 전부 불필요** — 웹·네이티브 같은 코드로 간다 |
| 응답 형태 | `resultCount` + `results[]`. `trackId`(number)·`trackName`·`artistName`·`trackTimeMillis`·`artworkUrl100` 전부 존재 |
| `600x600bb` 치환 | **동작.** `100x100bb`(7KB) → `600x600bb`(153KB) 둘 다 200 |
| content-type | `text/javascript` (JSON인데) — §11 참고 |

검증 명령 (재현용):
```bash
curl -s -D - -o /dev/null -H "Origin: https://demo-muzik.vercel.app" \
  "https://itunes.apple.com/search?term=aespa&entity=song&country=kr&limit=3" | grep -i access-control
```

**아직 실측 안 한 것**: `search.list` 매칭 정확도(§4 정렬 가중치가 실제로 공식 음원을 상단에 올리는지)와
캐시 히트율. 둘 다 API 키가 필요해 M1 이후에만 측정 가능하다.

## 12. 선행 조건과 구현 순서

**M1 필수.** API 키는 Functions 전용이다(`유튜브연동설계.md` §1·§6). `search.list`를 부르려면 서버가 있어야 한다.

`demo/spark-web`(Spark 무료 플랜, 클라 직접 쓰기)에는 **넣을 수 없다.** 키를 클라이언트에 박는 것 외에
방법이 없고 그건 정책 체크리스트 위반이다. 이 브랜치는 링크 붙여넣기 경로로 남는다.

**M2-b(`registerTrack`이 Functions callable로 교체되는 시점)에 함께 태운다.** 반대 순서면 스텁을 두 번 만든다.
검색이 최종적으로 하는 일은 videoId 하나를 기존 `registerTrack`에 넘기는 것뿐이라,
그 교체가 끝나 있으면 검색은 파이프라인 앞단에 화면 하나 붙이는 작업이다.

### Functions 추가 명세 (`백엔드설계.md` §3에 반영)

| 함수 | 입력 | 처리 | 출력 |
|---|---|---|---|
