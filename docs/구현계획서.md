# MUZIK IS MY LIFE — MVP 구현 계획서 (v2)

- 스택: **React Native (Expo) + Firebase** / 디자인: Figma 실측값 반영 완료
- 범위: MVP (방 생성·초대, 닉네임 입장, 유튜브 링크 파싱 등록(하루 1곡), 데일리 테마, 유튜브 핸드오프 재생, 자정 마감·과거 목록)
- v2 변경: ①마감 자정→새벽 4시 ②테마 서버리스(내장 풀) ③6자리 초대 코드 병행 ④재생을 유튜브 핸드오프 중심으로 전환 (2026-07-12 확정)
- v3 변경(2026-07-17): 마감을 **새벽 4시 → 자정(24시)로 환원**. `DAY_CUTOFF_HOUR=0` = KST 달력 날짜.

---

## 1. 아키텍처

```
[Expo RN 앱 (iOS)]
 ├─ UI: Figma 기반 화면 (expo-router) — 완료
 ├─ 재생(메인): watch_videos URL 핸드오프 → 유튜브 앱/웹에서 연속 재생
 ├─ 재생(보조): react-native-youtube-iframe 인앱 미리듣기
 ├─ 테마: 내장 테마 풀 + dateKey 해시 결정론적 선택 (서버 불필요)
 └─ 딥링크: muzik:// 스킴 + 6자리 초대 코드 입력 병행

[Firebase]
 ├─ Auth: Anonymous Auth (닉네임만으로 입장)
 ├─ Firestore: rooms / members / tracks (themes 컬렉션 삭제)
 ├─ Cloud Functions: registerTrack — 서버 시각으로 dateKey 확정 + 하루 1곡 검증
 │   (곡 메타데이터는 클라이언트에서 oEmbed 직접 호출 — 쿼터·키 불필요)
 └─ Security Rules: 문서 ID {uid}_{dateKey} + create-only로 하루 1곡 강제
```

**핵심 설계 결정**

1. **"음악적 하루"는 자정(24시, KST)에 끝난다.** `dateKey` = KST 달력 날짜(`DAY_CUTOFF_HOUR=0`). 마이그레이션 배치 없음 — "오늘 목록" = `dateKey == todayKey()` 쿼리. 등록 시각의 dateKey는 Functions 서버 시각으로 확정(클라이언트 시계 조작 방지). ※ 마감 시각은 상수 하나로 조절 가능(과거 새벽 4시였음).
2. **하루 1곡 = 문서 ID로 강제.** 트랙 문서 ID `{uid}_{dateKey}` + Rules create-only → 중복 등록 원천 차단.
3. **재생 = 유튜브 핸드오프.** `https://www.youtube.com/watch_videos?video_ids=ID1,ID2,…` 를 열면 무인증·무쿼터로 임시 재생목록(TLGG…)이 생성되어 유튜브 앱/웹에서 그대로 연속 재생됨(≤50곡, 2026-03 동작 확인). 인앱 IFrame은 '미리듣기' 보조로 유지 → 화면 꺼짐 정지·심사 리스크가 핵심 경험에서 제거됨.
4. **Data API(playlists.insert)는 영구 제외.** 핸드오프로 생성되는 TLGG 임시 재생목록에 유튜브가 자체 [저장] 버튼을 제공하므로, "내 계정에 저장"은 이미 무료로 해결됨 → 유료 기능 가치 없음. Data API는 `videos.list`(등록 검증, API 키만, 1유닛/호출)만 사용. 상세: `docs/유튜브연동설계.md`

## 2. 데이터 모델 (Firestore)

**스키마 정본은 `docs/백엔드설계.md` §2다.** (같은 정보를 두 문서에 두었더니 갈라졌다 — 여기선 포인터만 유지)

- 초대: ①`muzik://r/{code}` 딥링크 ②앱 내 6자리 코드 직접 입력 — 도메인/Universal Link 없이 베타 가능. 정식 런칭 때 `https://<도메인>/r/{code}` Universal Link 추가.
- **테마 진실원 (2026-07-13 확정)**: 곡이 0개인 날은 `days/{dateKey}` 문서가 아직 없는데 배너는 등록 **전에도** 떠야 한다. 그래서 **오늘 테마 = `themeFor(todayKey())` 클라 계산**. 반면 **과거 테마 = `days.themeText`**(없으면 `themeFor` 폴백) — THEMES 풀이 나중에 바뀌면 해시 결과가 달라져 과거 기록의 테마가 소급 변조되기 때문. `registerTrack`은 그날 days 문서를 **처음 만들 때만** themeText를 기록하고 이후 덮어쓰지 않는다.
- '추천 완료' 상태: 오늘 tracks에 내 uid 문서 존재 여부로 파생.

## 4. 남은 구현 순서

| 주차 | 작업 | 완료 기준 |
|---|---|---|
| 2주 | 익명 로그인, 방 생성/입장, inviteCode 발급·역참조, 딥링크+코드 입장 | 코드/링크로 타 기기 입장 성공 |
| 3주 | 곡 등록: oEmbed 파싱 미리보기 → registerTrack 함수(서버 dateKey) → Rules, 실시간 리스트(onSnapshot) | 하루 2곡 등록이 서버·룰에서 거부됨 |
| 4주 | 재생 완성: 핸드오프 URL(당일 order 순), 미리듣기 큐(ended→다음 곡), 자정 롤오버 검증 | 유튜브 앱에서 전곡 연속 재생 확인 |
| 5주 | 과거 목록(날짜 탭 ↔ dateKey), 곡 수정/삭제(당일 본인만), 마감 카운트다운 마감 직후 상태 전환 | 00:00 경과 시 목록 전환·보관 확인 |
| 6주 | QA: 링크 엣지케이스(shorts/si=/music.youtube), watch_videos 폴백, 동시 등록 → TestFlight | 기획서 QA 항목 통과 |

## 5. 리스크 및 대응

| 리스크 | 대응 |
|---|---|
| watch_videos 비공식 엔드포인트 (변경 가능성) | **3단계 폴백 체인** (Data API는 대안이 아니다 — §1-4): ①`watch_videos` 핸드오프(메인) ②`handoffMode='first_video'` — 첫 곡만 유튜브로 열고 인앱 미리듣기를 '연속 재생'으로 승격 ③완전 차단 시 — 인앱 IFrame 연속 재생이 메인, 유튜브는 곡별 딥링크. **최악의 경우에도 앱은 자체 재생으로 살아남는다.** M3에서 킬스위치 전환 테스트 |
| App Store 심사 | 핵심 재생이 유튜브 앱 핸드오프라 "유튜브 대체 앱" 리스크 대폭 완화. 미리듣기는 공식 IFrame 사용 명시 |
| 링크 형태 다양성 | extractVideoId 정규식(watch/youtu.be/shorts/embed/music) + oEmbed 폴백, 유닛 테스트 |
| 자정 경계 동시성 (23:59 등록) | registerTrack이 서버 시각으로 dateKey 확정 — 클라이언트 표기와 불일치 시 응답으로 교정 |
| 웹 미지원 (기획서 '링크만으로 참여'와 간극) | 보류 중 — react-native-web 추가 시 동일 코드로 참여자용 웹 제공 가능(제안 #1, 미확정) |

## 6. MVP 제외 (고도화 단계)

이모지 리액션, 채팅, 푸시 알림, 외부 플랫폼 변환(스포티파이/애플뮤직·단건 결제), 테마 스킨 판매, 아카이브 이미지 카드 공유(바이럴). ~~유튜브 계정 저장 유료 기능~~은 TLGG 무료 저장으로 대체되어 폐기. tracks에 reactions 서브컬렉션 확장 여지만 예약. YouTube 연동 상세는 `docs/유튜브연동설계.md`.
