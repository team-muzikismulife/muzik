# Firebase 백엔드 설계 (2주차)

원칙: **쓰기는 전부 Cloud Functions 경유, 클라이언트는 읽기 전용.**
하루 1곡·서버 시각 dateKey·초대 코드 유일성 같은 핵심 규칙을 클라이언트가 우회할 수 없게 만드는 가장 단순한 방법. Security Rules는 "읽기 권한 관리"에만 집중.

---

## 1. 인증 및 입장 흐름

```
앱 최초 실행 → signInAnonymously() → uid 발급 (기기 단위 고정)
방 입장 시 → 닉네임 입력 → members/{uid}에 방별로 저장
```

- 닉네임은 **방별** 속성 (같은 사람이 방마다 다른 닉네임 가능 — 기획서의 "입장 시 닉네임 입력"과 일치)
- 익명 계정 한계: 앱 삭제 시 uid 소멸 → 방 기록과의 연결 끊김. **MVP 수용**, 고도화 때 Sign in with Apple을 `linkWithCredential`로 연결(기존 uid 유지) 여지만 남김

> **향후 전환 예정 (2026-07-14 합의).** 소셜 로그인을 붙이면 닉네임을 **provider의 `displayName`에서 받아
> 계정 단위 프로필로** 쓴다 — 팀을 만들 때마다 이름을 다시 입력받지 않는다.
> 그때 `users/{uid}` 프로필 문서를 두고 `createRoom`/`joinRoom`의 `nickname` 인자를 없앤다(서버가 프로필에서 읽음).
> `members.nickname`은 **표시용 스냅샷으로 유지**한다 — 멤버를 그릴 때마다 프로필을 N번 읽지 않기 위해서다.
> 지금은 로그인이 없어 `displayName`이 존재하지 않으므로, 그때까지 팀 개설 화면에서 닉네임을 받는다.
- iOS 재설치 대비 최소 장치: 마지막 참여 방 코드들을 로컬(AsyncStorage)에 백업 → 재설치 시 코드로 재입장 유도

## 2. Firestore 스키마 (확정)

```
config/app                            # 킬스위치 등 원격 설정 (콘솔 수동 관리)
  handoffMode: 'watch_videos' | 'first_video'

rooms/{roomId}
  name, inviteCode, createdAt, createdBy(uid), memberCount

invites/{inviteCode}                  # 코드 → 방 역참조 (유일성 보장 지점)
  roomId

rooms/{roomId}/members/{uid}
  uid, nickname, joinedAt, photoColor   # uid는 문서 ID와 중복 — collectionGroup 쿼리에 필요하다

rooms/{roomId}/tracks/{uid_dateKey}
  videoId, title, artist, comment(≤30자),
  uid, nickname, dateKey, order(서버 epoch), createdAt,
  embeddable, durationSec, metaRefreshedAt, unavailable?

rooms/{roomId}/days/{dateKey}         # 날짜별 집계 (과거 목록 탭의 데이터 소스)
  dateKey, trackCount, coverVideoId(첫 곡), themeText, updatedAt
```

**썸네일 URL은 필드로 두지 않는다 (2026-07-13 변경).** `videoId`에서 파생한다(`src/lib/youtube.ts`).
유튜브연동설계 §1의 "videoId만이 영구 원본" 원칙의 당연한 귀결인데, 스키마가 이를 어기고 있었다.
URL을 따로 저장하면 videoId와 어긋날 수 있고 **실제로 어긋났다** — 목업에서 곡 제목과 배경 이미지가
서로 다른 곡을 가리켰다. `days.coverThumbnailUrl`도 같은 이유로 `coverVideoId`로 바꾼다.

**`themeText`는 '그날 첫 곡이 올라온 순간의 테마 스냅샷'이다.** 첫 생성 때만 기록하고 이후 upsert에서 덮어쓰지 않는다.
오늘 테마는 클라가 `themeFor(todayKey())`로 계산하고(곡 0개면 days 문서가 없으므로), 과거 테마만 이 필드를 읽는다 — 상세는 `구현계획서.md` §2.

**days가 필요한 이유**: Firestore는 "tracks에서 존재하는 dateKey 목록"을 뽑을 수 없음(distinct 불가). 첫 곡 등록 때 Functions가 days 문서를 만들어 날짜 탭·과거 목록·대표 썸네일을 쿼리 1번으로 해결.

**`members.uid`가 필드로도 있는 이유 (2026-07-14 추가).** "내가 참여 중인 방 목록"을 서버에서 뽑으려면
`collectionGroup('members').where('uid','==',myUid)`가 유일한 방법이다. 문서 ID로는 collectionGroup을 쿼리할 수 없다.
최초 계획(§7)은 이 목록을 **AsyncStorage 로컬 백업**으로 우회했으나, 앱 삭제·기기 변경 시 목록이 통째로
사라지는 두 번째 실패 모드가 생긴다 — 그래서 **서버를 진실로** 바꿨다. 상세: `온보딩구현계획.md` §1.

**인덱스**: tracks `dateKey ASC + order ASC` 복합 인덱스 1개. days는 문서 ID(=dateKey) 역순 조회라 불필요.
collectionGroup `members` — `uid ASC` 인덱스 1개 (참여 중인 팀 목록).

## 3. Cloud Functions (callable, asia-northeast3)

| 함수 | 입력 | 처리 | 출력 |
|---|---|---|---|
| `createRoom` | name, nickname | 코드 생성(6자, 혼동문자 IO01 제외 32자셋) → `invites/{code}` 트랜잭션 create로 유일성 확보(충돌 시 재생성) → rooms + members(방장) 생성 | roomId, inviteCode |
| `joinRoom` | code, nickname | invites 조회(없으면 not-found) → memberCount ≥ 30이면 거부 → members/{uid} set(재입장 시 닉네임 갱신) + memberCount 증가 | roomId |
| `registerTrack` | roomId, videoId, comment | 멤버 검증 → 서버 시각 dateKey(자정 컷) → videos.list 검증(유튜브연동설계 §2) → tracks/{uid_dateKey} **create**(존재 시 already-exists → 하루 1곡) → days upsert(trackCount+1, 첫 곡이면 cover·theme 기록) | track 스냅샷 |
| `updateTrack` | roomId, comment 또는 videoId | 본인+당일(dateKey==오늘)만 → videoId 변경 시 재검증 | track 스냅샷 |
| `deleteTrack` | roomId | 본인+당일만 → delete + days.trackCount−1 (**0이 되면 days 문서 삭제** — 안 그러면 빈 날짜 탭이 영원히 남는다) | ok |
| `refreshMeta` | roomId, dateKey | `metaRefreshedAt` 30일 경과 곡만 oEmbed 재조회(0유닛) → 제목 갱신 / 404면 `unavailable: true` | 갱신된 트랙 목록 |

**`refreshMeta`가 필요한 이유**: 메타데이터 30일 lazy refresh(유튜브연동설계 §5)는 **쓰기**다. 클라이언트는 Firestore에 쓸 수 없으므로(§ 대원칙) Functions여야 한다. 이 문서의 최초 5개 명세에서 누락되었던 것 — 2026-07-13 추가.

- 전부 `request.auth` 필수 (익명 토큰). App Check(기기 무결성)는 소프트 런칭 때 활성화
- 코드 충돌 확률: 32^6 ≈ 10억 조합, 방 1만 개 시 충돌 재시도 확률 무시 가능

## 4. Security Rules (초안)

```
match /config/app                     { allow read: if true; }
match /rooms/{roomId} {
  allow read: if isMember(roomId);
  match /members/{uid}                { allow read: if isMember(roomId); }
  match /tracks/{trackId}             { allow read: if isMember(roomId); }
  match /days/{dateKey}               { allow read: if isMember(roomId); }
}
match /invites/{code}                 { allow read: if request.auth != null; }

// 내 멤버 문서는 roomId를 몰라도 읽을 수 있어야 한다 (참여 중인 팀 목록 = collectionGroup 조회)
match /{path=**}/members/{memberId} {
  allow read: if request.auth != null && resource.data.uid == request.auth.uid;
}

// 쓰기는 전 컬렉션 deny — Admin SDK(Functions)만 통과

function isMember(roomId) {
  return request.auth != null &&
    exists(/databases/$(database)/documents/rooms/$(roomId)/members/$(request.auth.uid));
}
```

주의: `isMember`는 문서 read당 exists() 1회 → 비용 있음. 리스트 쿼리는 컬렉션 단위 1회 평가라 MVP 규모에서 무시 가능.

주의: collectionGroup 룰이 `resource.data.uid`를 보므로 **클라이언트 쿼리도 `where('uid','==',myUid)`로 같은 제약을
걸어야** 통과한다. 룰과 쿼리가 어긋나면 `permission-denied`가 나는데, 원인이 룰인지 쿼리인지 구분하기 어렵다.

## 5. 클라이언트 실시간 구독

```
[홈 화면 - 포커스 중에만 구독 (useFocusEffect)]
  tracks: where dateKey == todayKey() orderBy order  → onSnapshot
  members: 전체                                       → onSnapshot
  days: orderBy(문서ID desc) limit 14                 → onSnapshot (날짜 탭)

[상세 화면]
  해당 dateKey의 tracks 1회 get (과거) / onSnapshot (오늘)
```

- 자정 롤오버: 홈에서 카운트다운이 0 도달 시 todayKey() 재계산 → 쿼리 재구독 (앱 켜둔 채 넘어가는 경우 대응)
- 오프라인: Firestore 기본 캐시 사용, 등록류는 Functions라 오프라인 시 명시적 에러 → "네트워크 연결 후 다시 시도" 토스트
- 상태관리: zustand 스토어 2개 — `session`(uid, 최근 roomId, 로컬 백업 코드), `room`(members, todayTracks, days, theme)

## 6. 엣지케이스

| 케이스 | 처리 |
|---|---|
| 23:59:59 등록 vs 00:00 롤오버 | 서버 시각이 유일 기준. 클라 표기와 다르면 registerTrack 응답의 dateKey로 UI 교정 |
| 같은 코드 동시 joinRoom | members set은 멱등. memberCount는 트랜잭션 증가 |
| 방장 이탈 | MVP에선 방장 개념 최소화(생성자 표기만). 방 삭제 기능 없음 → 고도화 |
| 초대 코드 무차별 대입 | 코드 32^6 공간 + Functions rate limit(App Check 후 강화) |
| 닉네임 중복 | 허용 (uid로 구분, 아바타 색으로 시각 구분) |

## 7. 개발 환경·구현 순서 (M1)

> 일정은 주차가 아니라 **마일스톤 M0~M5**로 끊는다 — "실기기에서 무엇을 보여줄 수 있는가" 기준.
> 상태 UI·접근성·에러 처리는 6주차 QA에 몰지 않고 **M0에 선불로 깔았다**(완료). 그러지 않으면
> 매 단계가 "보여줄 순 있지만 로딩하면 흰 화면"인 데모가 된다. 전체 마일스톤은 `CLAUDE.md` 참조.

**M1 = "두 기기가 같은 방에 있다"** (PR 단위)

0. Firebase 프로젝트 생성, Anonymous Auth on, **Emulator Suite**(auth/firestore/functions)로 로컬 개발 — 실 프로젝트 없이도 전체 플로우 테스트 가능
1. `functions/` 워크스페이스 셋업(TypeScript) + createRoom/joinRoom + 에뮬레이터 테스트
2. 클라: 익명 로그인 부트스트랩 + 온보딩 실연동(방 리스트 = **collectionGroup `members` 조회**) → `온보딩구현계획.md`
3. 초대: 코드 입력 입장 + `muzik://r/{code}` 딥링크 → joinRoom → 방 홈 이동
4. Security Rules 배포 + 룰 단위 테스트(@firebase/rules-unit-testing)
5. 홈 화면 onSnapshot 연결(members/days) — tracks는 M2 registerTrack과 함께

완료 기준: 기기 A에서 방 생성 → 코드 전달 → 기기 B 입장 → 양쪽 멤버 리스트 실시간 반영.

### M1에서 막힐 지점 (미리 알고 들어간다)

| 함정 | 대응 |
|---|---|
| **RN Auth 영속** — `getAuth()`는 RN에서 재시작 시 로그아웃된다. **uid 소멸 = 방 소유권 소멸.** | `initializeAuth(app, { persistence: getReactNativePersistence(AsyncStorage) })`. firebase v11에서 타입이 누락돼 `@ts-expect-error`가 필요할 수 있다(v10의 `firebase/auth/react-native` 서브패스는 v11에서 제거됨) |
| **에뮬레이터 접속** — Expo Go 실기기는 `localhost`를 못 본다 | `Constants.expoConfig.hostUri`에서 개발 PC의 LAN IP를 파싱해 `connect*Emulator`에 넘기는 `src/lib/emulator.ts` 필수. + Windows 방화벽 인바운드(9099/8080/5001) |
| **Functions 리전** | 클라에서 `getFunctions(app, 'asia-northeast3')`를 빠뜨리면 us-central1로 호출돼 `internal` 에러 — 원인 찾기 어렵다 |
| **Firestore 스트리밍** | `experimentalForceLongPolling` 없으면 Expo Go에서 onSnapshot이 조용히 멈추는 사례 |
| **딥링크 검증** | `muzik://`는 **Expo Go에서 검증 불가**(`exp://…/--/r/CODE`로만 열림). `expo-dev-client` 1회 빌드를 M1에 끼우는 것을 권장. 공유 텍스트엔 링크와 **6자 코드를 반드시 병기** |
